# Runbook

Operational procedures for stumblestack. The system is static (git + GitHub
Pages + the MCP server clients run themselves), so most "operations" are recovery
and the deliberately-deferred resilience features.

## Recovery (DR)

The entire system is reconstructable from any clone:

```bash
git clone https://github.com/Wisdoverse/stumblestack
cd stumblestack
python3 scripts/build_index.py     # regenerate index.json
python3 scripts/build_site.py      # regenerate the site into _site/
```

Pages redeploys from `main` via `.github/workflows/pages.yml`. `index.json` is
regenerable, so a corrupted copy is never fatal.

## Custom domain / HTTPS

`stumblestack.dev` is a Cloudflare-DNS-only apex pointing at GitHub Pages' anycast
IPs (`185.199.108-111.153` + the matching AAAA). The cert is Let's Encrypt, issued
by GitHub. If HTTPS breaks:

1. Confirm DNS resolves to the GitHub IPs (`dig +short stumblestack.dev`), DNS-only
   (grey cloud) on Cloudflare — an orange-cloud proxy blocks GitHub's ACME challenge.
2. Repo → Settings → Pages → "Check again" to re-trigger certificate issuance.
3. Re-enable "Enforce HTTPS".

## Deferred resilience features (tracked, not yet shipped)

These were scoped in the v1.0 design review and intentionally NOT shipped as
workflows, because they require external infrastructure that cannot be verified
in CI without secrets/credentials. Each has a tracking issue; implement when the
trigger fires.

### Read-only mirror to a second forge (trigger: GitHub availability concern)

Goal: a warm mirror on Codeberg/GitLab so a GitHub outage or account dispute does
not dark the corpus.

Procedure (manual until automated):
1. Create an empty mirror repo on the second forge.
2. Add a deploy key / token as a repo secret `MIRROR_SSH_KEY`.
3. `git push --mirror <second-forge-url>` on a schedule.
4. Failover: point the `stumblestack.dev` CNAME (or a Cloudflare Worker) at the
   mirror's Pages equivalent. Consumers can also set `STUMBLESTACK_MIRRORS` to the
   mirror base URL as an immediate stopgap (the MCP server already supports a
   comma-separated mirror chain with automatic failover).

### Signed index (trigger: consumers need origin assurance)

Goal: cosign-sign `index.json` (keyless, GitHub OIDC) and publish the bundle so
consumers can verify provenance without trusting raw.githubusercontent.com.

Procedure: a `workflow_dispatch` + tag-gated workflow runs `cosign sign-blob
--bundle` and attaches the bundle to the GitHub Release. Resolve and SHA-pin
`sigstore/cosign-installer` before enabling (per the SHA-pin policy). Consumers
verify with `cosign verify-blob --bundle ... index.json` against the identity
`.../sign-index.yml@refs/tags/v*`.

### Update webhooks (trigger: a consumer asks for push instead of poll)

Goal: POST `{event, entries_added:[uuid]}` to subscriber URLs (loaded from a public
`webhooks.txt`) on each `main` push. URLs are validated as public https (the same
rules as link validation); per-URL failures are isolated; first run with an empty
list is a no-op.

### Embeddings backend (trigger: lexical recall proves insufficient)

Goal: an optional `embeddings.json` artifact (generated by a key-gated
`workflow_dispatch` job) that the MCP server prefers over lexical search when
present, with graceful fallback to lexical when absent. The seam is documented in
DESIGN.md §8; build it when the eval set shows lexical recall is the bottleneck.
